# TRADINGLAB — Backtesting Engine Operations (OPS)

The **Backtesting Engine** is the high-performance simulation core of TRADINGLAB. It transforms abstract strategy logic (generated by the Lab Assistant) into verifiable performance data using historical market dynamics.

---

## 1. Strategy Ingestion Flow

The engine consumes the **JSON Strategy Contract** produced by the Lab Assistant.

### Ingestion Pipeline:
1. **Contract Validation**: Ensures the JSON contains required fields (`ticker`, `timeframe`, `logic`, `parameters`).
2. **Logic Parsing**: Deconstructs the `logic` string into executable conditional triggers.
3. **Environment Setup**: Initializes the starting capital, slippage models, and commission structures defined in the UI.

> [!TIP]
> **Full Technical Reference**: For a complete list of supported indicators (SMA, RSI, MACD, etc.), logic operators, and JSON schemas, see: [backtesting-supported-features.md](file:///c:/Users/User/OneDrive/Desktop/TRAIDINGLAB%20V8/my-app/docs/backtesting-supported-features.md).

---

## 2. Core Simulation Model

Our engine uses a **Vectorized & Event-Driven Hybrid** approach for maximum speed without sacrificing tick-level accuracy.

### Execution Mechanics:
- **OHLCV Iteration**: The engine steps through historical candles sequentially.
- **Signal Processing**: At each step, the strategy logic is evaluated against the current (and historical) data state.
- **Order Modeling**: 
    - **Slippage**: Configurable basis-points (BPS) adjustment to fill prices.
    - **Commissions**: Tiered fee structure support (Maker/Taker).
    - **Position Sizing**: Supports fixed-pip, percentage-of-equity, or Kelly Criterion sizing.

---

## 3. Data Architecture

The engine relies on a dual-layer data fetching strategy to ensure zero latency during repeated tests.

- **Layer 1: Alpha Vantage Cloud**: Primary source for high-fidelity OHLCV data.
- **Layer 2: Supabase Cache**: Every unique ticker/timeframe request is cached as a JSONB blob in the `market_data_cache` table to avoid redundant API consumption and quota exhaustion.

---

## 4. Analytics & Risk Metrics

Beyond the Equity Curve, the engine calculates institutional-grade risk metrics:

| Metric | Description |
| :--- | :--- |
| **Sharpe Ratio** | Risk-adjusted return relative to volatility. |
| **Max Drawdown (MDD)** | The largest peak-to-trough decline in equity. |
| **Win/Loss Ratio** | Frequency of profitable vs. losing trades. |
| **Profit Factor** | Gross Profit / Gross Loss. |

---

## 5. Visualization Layer (UX)

- **Dynamic Equity Curve**: SVG-based pathing that updates in real-time as the simulation progresses.
- **Trade Scrubbing**: Interactive timeline that allows users to hover over trade markers and see exact entry/exit conditions on the chart.
- **Micro-Animations**: Uses `framer-motion` for smooth stat updates and "Glow-Pulse" effects on high-conviction trades.

---

## 6. Integration: Lab -> Backtest -> Live

1. **Lab**: Researcher designs the "Technical Hypothesis".
2. **Backtest**: Engine validates the hypothesis against 1, 5, or 10 years of data.
3. **Deploy**: (Future) One-click transition from a successful backtest to a Paper Trading or Live Execution bot.

---

> [!IMPORTANT]
> The engine is designed to be **Deterministic**. Running the same strategy on the same data with the same parameters will *always* yield identical results, ensuring scientific rigor.
